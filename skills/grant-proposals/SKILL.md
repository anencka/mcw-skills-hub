# Grant & Proposal Writing

## When to Use
Developing or revising a funding proposal: specific aims, significance, innovation, approach,
preliminary data narrative, or responses to reviews.

## First-use setup (read the profile; fill if thin)
Read `~/work-profile/current-projects.md` (the project, its funding/sponsor and status),
`~/work-profile/goals-and-priorities.md` (what the operator is optimizing for), and
`~/work-profile/domain-knowledge.md` (field + review/funding dynamics). If any is still mostly
`*(to capture)*`, offer **once** to fill it — a few questions (hand to `onboarding-interview`,
e.g. "let's do current-projects") or the operator edits it in the Files tab — then proceed; if
they skip, note that in memory and don't re-ask. **Read these fresh each use; never cache their
contents — the file is the source of truth.** Funder/mechanism rules, the central hypothesis, and
available preliminary data are proposal-specific — gather those for the task at hand.

## Procedure
1. **Start from the criteria.** Map the funder's required sections and review criteria; write to
   what reviewers score, in their order.
2. **Nail the Specific Aims page first** — it's the proposal in miniature: problem/gap → long-term
   goal → central hypothesis → 2–3 aims (each a testable objective, not a method) → payoff.
3. **Significance & Innovation:** argue importance and what's genuinely new; avoid overclaiming —
   reviewers punish hype.
4. **Approach:** for each aim, rationale → design → expected outcomes → pitfalls & alternatives.
   Tie to preliminary data the operator actually has; mark gaps as `[NEED DATA/CITATION]`.
5. **Reviewer lens:** after drafting, read as a skeptical reviewer — feasibility, rigor,
   over-ambition, missing controls — and list the weaknesses to fix.
6. **Tighten** to limits without gutting the argument; keep formatting/compliance correct.

## Pitfalls
- Aims that are methods ("we will run X") rather than objectives ("we will determine whether…").
- Overclaiming significance/innovation, or feasibility not backed by preliminary data.
- Ignoring the funder's exact structure, limits, or review criteria.
- Inventing preliminary results or citations — never.

## Verification
Every required section is present and within limits; aims are objectives with a clear hypothesis;
claims map to real data/citations or are flagged; a skeptical-reviewer pass has been done and its
findings addressed.
